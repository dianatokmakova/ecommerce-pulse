# E-Commerce Pulse — техническая спецификация проекта

**Сквозная аналитическая платформа для e-commerce: от сбора данных до дашборда с детекцией аномалий цен.**

Автор: Aslan · Версия: 1.0 · Дата: 2026-06-26

---

## 0. TL;DR (за 30 секунд)

Платформа построена вокруг **одного склада данных (ClickHouse)**, в который текут два потока:

1. **Исторический e-com (Olist)** — реальные 100k заказов бразильского маркетплейса. На них считается классическая аналитика: выручка, AOV, RFM-сегментация, когортный retention, LTV, SLA доставки, анализ отзывов.
2. **Живой поток цен** — раз в час Airflow дёргает API, кладёт сырьё в MinIO (S3), грузит в ClickHouse и ищет **аномалии цен** (z-score / IsolationForest). Источник цен — **собственный «фейковый маркетплейс»**, развёрнутый на Vercel: это легально, управляемо и позволяет подмешивать аномалии в данные.

Поверх склада — **два BI-слоя**: публичный дашборд на **Yandex DataLens** (бесплатно) и **собственный дашборд** на ECharts (Vercel) поверх тонкого read-only API (`dashboard-api`).

Всё работает в **Docker** на сервере. Деплой полностью бесплатный.

## 1. Проектные решения и их обоснование

| Задача / требование | Как решается в проекте |
|---|---|
| Аналитический SQL (не `select * limit 10`) | Оконные функции, RFM, когорты, LTV на ClickHouse |
| ClickHouse как аналитическая СУБД | Весь склад на ClickHouse: bronze/silver/gold, движки `MergeTree`, `ReplacingMergeTree`, чтение из S3 |
| Python шире pandas | Ingestion-скрипты (httpx), генерация данных, ML-аномалии (sklearn), свой API на FastAPI |
| BI и визуализация | Публичный дашборд на Yandex DataLens + собственный дашборд на ECharts (Vercel) поверх `dashboard-api`; оба читают gold-витрины ClickHouse |
| Автономность / инженерия | Airflow оркестрирует пайплайны по расписанию, есть data-quality проверки |
| Бизнес-ориентированные метрики | Метрики и аномалии, которые реально нужны e-com: динамическое ценообразование, спрос, отток |
| Прод-готовность | Docker, бесплатный деплой, README с архитектурой, публичные ссылки на дашборды |

Проект собран как **мини-копия реального дата-отдела**: источник → оркестратор → озеро (lake) → склад (warehouse) → витрины (marts) → BI. Используются те же инструменты и роли, что и в реальном дата-отделе.

**Медальон-архитектура (bronze → silver → gold)** — индустриальный стандарт:

- **gold (marts)** — витрины под конкретные вопросы бизнеса/дашборда (агрегаты, RFM, когорты).

## 4. Источники данных

### 4.1. Исторический e-com — Olist (batch)
Реальный публичный датасет **Brazilian E-Commerce Public Dataset by Olist** (Kaggle): ~100 000 заказов 2016–2018, 9 CSV-таблиц:

| Таблица | Что внутри |
|---|---|
| `olist_orders_dataset` | заказы, статусы, таймстемпы (покупка → доставка) |
| `olist_order_items_dataset` | позиции заказов, цена, стоимость доставки |
| `olist_order_payments_dataset` | платежи, тип, число рассрочек |
| `olist_order_reviews_dataset` | отзывы и оценки 1–5 |
| `olist_customers_dataset` | клиенты + гео (zip, город, штат) |
| `olist_products_dataset` | товары, категория, габариты |
| `olist_sellers_dataset` | продавцы + гео |
| `olist_geolocation_dataset` | zip → lat/lng |
| `product_category_name_translation` | перевод категорий PT→EN |

На этих данных — вся «глубокая» e-com аналитика (раздел 7).

### 4.2. Живой поток цен — Fake Marketplace API (streaming-ish)
Парсинг реальных маркетплейсов (Ozon, Wildberries, Яндекс Маркет) недопустим — это нарушение ToS и юридический риск, плюс активные баны. Поэтому используется **собственный API** на Vercel (`fake-marketplace-api/`), который:

- отдаёт корзину из ~50 товаров с ценами и остатками;
- цены **дрейфуют** во времени (случайное блуждание + сезонность по часам/дням);
- раз в N запросов **подмешивает аномалию** (скачок цены +40%, обнуление остатка) для срабатывания детектора аномалий.

Свой источник: легально, управляемо, реализует сценарий «динамическое ценообразование + поиск аномалий».

### 4.3. Внешнее обогащение (реальные бесплатные API без ключа)
Интеграция с внешними сторонними API:

- **Frankfurter** (`api.frankfurter.app`) — курсы валют без ключа. Конвертация Olist BRL → USD/RUB, обогащение цен.
- **Open-Meteo** (`api.open-meteo.com`) — погода без ключа и лимитов. Гипотеза «погода влияет на спрос» → фича для анализа.
- **CoinGecko** (keyless demo) — опционально, как ещё один «дрейфующий» ценовой ряд.

## 5. Что/где/как подключать в Docker (подробно)

Полный `docker-compose.yml` лежит в корне репозитория. Здесь — что каждый сервис делает и как соединяется.

### 5.1. Сервисы и порты

| Сервис | Внутренний адрес | Порт наружу | Логин по умолчанию |
|---|---|---|---|
| ClickHouse | `clickhouse:8123` (HTTP), `:9000` (native) | 8123 / 9000 | `analyst` / см. `.env` |
| MinIO | `minio:9000` (S3 API), `:9001` (консоль) | 9000 / 9001 | `MINIO_ROOT_USER` / `...PASSWORD` |
| Airflow webserver | `airflow-apiserver:8080` | 8080 | `admin` / см. `.env` |
| Airflow Postgres (метаданные) | `postgres:5432` | — (только внутри) | airflow / airflow |
| Caddy (опц.) | `:80`, `:443` | 80 / 443 | reverse-proxy |

> На одном сервере все сервисы общаются по **имени сервиса** внутри Docker-сети (`clickhouse`, `minio` и т.д.), а не по `localhost`. Это важно: в строке подключения Airflow→ClickHouse хост = `clickhouse`, а не `127.0.0.1`.

### 5.2. Как соединяются компоненты (потоки данных)

1. **Airflow → внешний источник.** DAG `live_pricing_ingest` (cron `@hourly`) httpx-ом тянет JSON из Fake API + Frankfurter, кладёт **Parquet в MinIO** (`s3://raw/pricing/dt=YYYY-MM-DD/hour=HH/...`).
2. **MinIO → ClickHouse.** ClickHouse читает файл напрямую через `s3()`-функцию и делает `INSERT INTO bronze.pricing_raw SELECT ... FROM s3(...)`. (Альтернатива для «стриминга» — движок `S3Queue`, который сам подхватывает новые файлы в бакете.)
3. **bronze → silver → gold.** SQL-задачи в Airflow пересобирают чистые таблицы и витрины (`INSERT INTO ... SELECT`, материализованные представления).
4. **Anomaly job.** Python-задача забирает свежие цены из ClickHouse в pandas, считает z-score / IsolationForest, пишет флаги обратно в `gold.pricing_anomalies`.
5. **Olist batch.** DAG `olist_batch_load` (cron `@daily`) грузит CSV → MinIO → ClickHouse `bronze.olist_*`, затем строит `silver`/`gold` витрины.
6. **BI читает только gold.** DataLens и свой дашборд (через `dashboard-api`) подключаются к ClickHouse и берут данные **только из gold-витрин** (быстро, безопасно, без тяжёлого SQL на лету).

### 5.3. Подключение ClickHouse ↔ MinIO
ClickHouse «видит» MinIO как S3. Пример чтения (см. `analytics/sql/`):

```sql
INSERT INTO bronze.pricing_raw
SELECT *
FROM s3(
  'http://minio:9000/raw/pricing/dt=2026-06-26/hour=10/data.parquet',
  '{{ minio_key }}', '{{ minio_secret }}',
  'Parquet'
);
```

Креды MinIO задаются один раз в `config.xml` ClickHouse как named-collection, чтобы не дублировать их в каждом запросе.

### 5.4. Подключение Airflow → ClickHouse
Подключение в Airflow задаётся через Connection (`Admin → Connections`) или env:
- тип: HTTP / generic, host `clickhouse`, port `8123`, схема `http`, логин/пароль из `.env`.
- В Python-задачах используется `clickhouse-connect` (официальный драйвер): `clickhouse_connect.get_client(host='clickhouse', port=8123, username=..., password=...)`.

### 5.5. Память сервера
Реалистичные минимумы по памяти:

- ClickHouse: 2 ГБ минимум, 4 ГБ комфортно.
- Airflow (LocalExecutor + Postgres): ~3–4 ГБ.
- MinIO: ~0.5 ГБ.

**Рекомендация: сервер от 8 ГБ RAM.** При 4 ГБ используется «lite»-профиль: ClickHouse + Airflow (LocalExecutor) + MinIO, а BI выносится в DataLens (облако, сервер не нагружает). Для ClickHouse при малой памяти задаётся `max_server_memory_usage_to_ram_ratio` и уменьшенный mark cache (см. `infra/clickhouse/`).

## 6. Модель данных (слои)

### bronze (сырьё, как пришло)
- `bronze.olist_orders`, `bronze.olist_order_items`, … (9 таблиц 1:1 с CSV, всё `String`/`Nullable`, минимум типизации).
- `bronze.pricing_raw` — сырые снимки цен (product_id, price, stock, captured_at, source).

### silver (очищенное)
- `silver.orders` — типизированные заказы, распарсенные даты, признак «доставлен/отменён».
- `silver.order_items` — позиции + связь с продуктом и продавцом.
- `silver.customers`, `silver.products`, `silver.sellers`, `silver.reviews`.
- `silver.pricing` — дедуплицированные цены (`ReplacingMergeTree` по `(product_id, captured_at)`), цена в BRL и USD (через Frankfurter).

### gold (витрины под дашборд)
- `gold.daily_revenue` — выручка, заказы, AOV по дням/категориям/штатам.
- `gold.rfm` — RFM-скоринг и сегмент каждого клиента.
- `gold.cohort_retention` — матрица удержания по когортам месяца первой покупки.
- `gold.delivery_sla` — план/факт сроков доставки, доля просрочек.
- `gold.review_stats` — распределение оценок, связь оценки с просрочкой.
- `gold.pricing_anomalies` — цены + флаг аномалии + метод + скор.

Ключевые движки ClickHouse: `MergeTree` (факты), `ReplacingMergeTree` (дедуп цен), `SummingMergeTree`/материализованные представления (предагрегаты), `ORDER BY` по дате/ключу для скорости.

## 7. Аналитика для e-commerce (что считаем и зачем)

Все витрины строятся SQL-ом на ClickHouse (примеры в `analytics/sql/`).

**Базовые метрики бизнеса**
- Выручка, число заказов, **AOV** (средний чек), доля доставленных/отменённых.
- Топ категорий и продавцов по выручке; вклад регионов (штатов).

**Поведение клиентов**
- **RFM-сегментация** (Recency, Frequency, Monetary) → сегменты «чемпионы», «спящие», «новички», «отток-риск». Готовый список «кого реактивировать».
- **Когортный анализ удержания**: по месяцу первой покупки смотрим, какая доля возвращается в месяц 1, 2, 3… Классическая heatmap-матрица.
- **LTV** (упрощённо): средняя выручка на клиента × срок жизни; сравнение LTV по каналам/категориям.

**Операционная аналитика**
- **SLA доставки**: план vs факт, доля просрочек, связь просрочки с регионом и с оценкой отзыва (гипотеза «просрочка роняет рейтинг»).
- Анализ отзывов: распределение оценок, доля 1–2★, корреляция с временем доставки.

**Динамическое ценообразование (живой поток)**
- Динамика цен по товарам/категориям, скользящие средние.
- **Детекция аномалий**: товар, у которого цена скакнула на 40% за сутки без причины, или обнулился остаток. Два метода:
  - статистика: z-score / межквартильный размах (IQR), правило 3σ;
  - ML: `IsolationForest` (scikit-learn), флаг `is_anomaly`.
- Alert-панель «аномалии прямо сейчас» — то, что реально полезно бизнесу.

## 8. BI-слой: два дашборда

### 8.1. Yandex DataLens (основной, публичный, бесплатный)
- **Почему он.** Бесплатен для небольших проектов (даже коммерческих). Подключается к **внешнему ClickHouse** напрямую.
- **Подключение.** В DataLens создаётся подключение типа ClickHouse: host = публичный адрес сервера, port `8443`/`8123`, включён HTTPS, логин/пароль read-only пользователя.
- **Безопасность.** При подключении к внешнему (не Yandex.Cloud) источнику нужно **добавить в whitelist диапазоны IP DataLens** на стороне ClickHouse/файрвола. Список IP — в документации DataLens.
- **Состав дашборда.** Чарты выручки, RFM-сегменты, когортная heatmap, карта по штатам Бразилии, alert-лист аномалий. Публикация через **DataLens Public** даёт публичную ссылку.
- Нужен аккаунт Yandex Cloud с привязанной картой (биллинг-аккаунт обязателен, но **использование в free-лимите бесплатно**).

### 8.2. Свой дашборд (ECharts на Vercel + `dashboard-api`)
- **Зачем.** Кастомный дашборд вместо коробочного BI: полный контроль над визуализацией при нулевой стоимости хостинга.
- **Фронт.** Один статический файл `dashboard/index.html` на ECharts, деплоится на Vercel (статика, без сборки).
- **Бэк.** Тонкий read-only слой `dashboard-api` (FastAPI) поверх gold-витрин ClickHouse: фиксированные запросы под read-only пользователя `bi_reader`. Поднимается вместе со стеком (`docker compose`, `:8000`); в проде закрыт HTTPS через Caddy, т.к. фронт на Vercel требует https.
- **Состав.** KPI выручки/AOV, динамика выручки и топ категорий/штатов, RFM-сегменты, когортная heatmap, SLA доставки, маржа по категориям, alert-лист ценовых аномалий — всё поверх gold-витрин.

> Оба BI-инструмента читают одни и те же gold-витрины ClickHouse: DataLens — публичный дашборд, свой дашборд на ECharts — второй (через `dashboard-api`).

## 9. Бесплатный деплой

| Слой | Где разворачиваем | Стоимость |
|---|---|---|
| ClickHouse, MinIO, Airflow | сервер, `docker compose up -d` | 0 (свой сервер) |
| Fake Marketplace API | Vercel (serverless, hobby) | 0 |
| Лендинг проекта | Vercel (static / Next.js) | 0 |
| Публичный дашборд | Yandex DataLens (free-лимит) | 0 |
| Свой дашборд (ECharts) | Vercel (фронт) + `dashboard-api` на сервере | 0 |
| Код, README, скриншоты | GitHub | 0 |
| HTTPS-домен | Let's Encrypt через Caddy (+ можно free-поддомен) | 0 |

Подробный пошаговый деплой — в `docs/DEPLOY.md`. Ключевые моменты:

- **Vercel ≠ хостинг для Airflow/ClickHouse.** На Vercel размещаются только статика/Next.js и serverless-функции (Fake API). Тяжёлый стек — на сервере. Python-дашборд (Streamlit) бесплатно хостится на **Streamlit Community Cloud** или **Hugging Face Spaces**, не на Vercel.
- **Безопасность (обязательно, раз ClickHouse смотрит в интернет для DataLens):**
  - отдельный **read-only** пользователь ClickHouse для BI;
  - **HTTPS** (Caddy + Let's Encrypt), не отдавать `8123` голым в интернет;
  - **whitelist IP** DataLens + доверенный IP; всё остальное — `deny` в файрволе (ufw);
  - консоли Airflow и MinIO — **не публиковать** открыто (либо basic-auth через Caddy, либо доступ по SSH-туннелю);
  - секреты — только в `.env` (он в `.gitignore`), в репозитории — `.env.example`.

## 12. Ключевые проектные решения

- Система самообновляется: Airflow по часам тянет данные, складывает в ClickHouse, ищет аномалии цен (а не разовый ноутбук).
- Выбран **ClickHouse, а не Postgres**: задача — аналитические агрегаты по миллионам строк, колоночное хранение даёт здесь выигрыш по скорости на порядок.
- Сырьё кладётся в **MinIO как в озеро**, в ClickHouse читается через `s3()` — медальон bronze/silver/gold.
- Аномалии ловятся двумя способами — **статистикой (z-score/IQR)** и **IsolationForest** — со сравнением по доле ложных срабатываний.
- Для отдела продаж собраны **RFM и когортный retention**: сегменты на реактивацию и точка проседания когорт.
